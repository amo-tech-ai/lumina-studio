# 1. `parallel_tool_calls` issue

Current code:

```ts
const hasTools =
  req.tools?.length ||
  (req.tool_choice && req.tool_choice !== "none") ||
  req.parallel_tool_calls;
```

This can classify the following as a tool request:

```ts
{
  model: "structured",
  parallel_tool_calls: true
}
```

even though there are no tool declarations. The current implementation then silently overrides the requested model tier.

Cloudflare documents `parallel_tool_calls` as controlling parallel function calling **during tool use**. It is not evidence that a request without tool declarations should automatically become a tool request. ([Cloudflare Docs][1])

### Correction to the proposed fix

The uploaded fix still treats this as tool-bearing:

```ts
tool_choice: "auto"
tools: undefined
```

A stronger solution is to validate the request first:

```ts
const declaredTools = (req.tools?.length ?? 0) > 0;
const toolHistory = req.messages.some(
  (message) =>
    message.role === "tool" ||
    (message.role === "assistant" &&
      (message.tool_calls?.length ?? 0) > 0),
);

const requestsToolSelection =
  req.tool_choice !== undefined &&
  req.tool_choice !== "none";

if ((requestsToolSelection || req.parallel_tool_calls) && !declaredTools && !toolHistory) {
  throw new ClientRequestError(
    "tool_choice and parallel_tool_calls require declared tools or an existing tool conversation",
  );
}

const needsToolProvider = declaredTools || toolHistory;
```

**Best practice:** reject internally inconsistent input with `400`; do not silently change the model.

# 2. Registry override fallback

This is confirmed.

The current resolver treats an override as a complete replacement:

```ts
export function getRegistry(overrides?: ModelRegistry): ModelRegistry {
  return overrides ?? DEFAULT_REGISTRY;
}
```

When `tool-calling` is absent from the override:

1. `resolveModelEntry("tool-calling", override)` returns `undefined`.
2. The router calls `resolveModelEntry("default")` without the override.
3. The built-in default is Gemini.
4. Gemini rejects the tool-bearing request.

### Red flag in Option B

Merging defaults and overrides is reasonable, but this code alone is insufficient:

```ts
{
  tiers: {
    ...DEFAULT_REGISTRY.tiers,
    ...overrides.tiers,
  }
}
```

Problems:

* `DEFAULT_REGISTRY` is currently not exported.
* Override JSON is only cast to `ModelRegistry`; it is not runtime-validated.
* An invalid override entry could replace a safe built-in entry.
* Empty or malformed fields could survive until provider execution.
* Security-sensitive tier fallback becomes implicit.

### Recommended design

```ts
function buildEffectiveRegistry(
  override: ModelRegistry | undefined,
): ModelRegistry {
  const validated = override
    ? modelRegistrySchema.parse(override)
    : undefined;

  return {
    tiers: {
      ...DEFAULT_REGISTRY.tiers,
      ...(validated?.tiers ?? {}),
    },
  };
}
```

Then fail closed for tool requests:

```ts
const entry = resolveModelEntry(modelTier, effectiveRegistry);

if (!entry) {
  throw new ClientRequestError(
    `No configured model supports tier "${modelTier}"`,
  );
}
```

Do **not** allow a tool request to fall back to a non-tool Gemini default.

# 3. Multi-turn tool continuation

This is a **production blocker**, not merely P2.

A normal tool loop is:

```text
User request + tools
→ assistant tool_calls
→ tool result message
→ model produces final answer
```

On the second request, clients may omit the original top-level `tools`. The current router then sees no tool flags and routes according to `req.model`, potentially returning to Gemini. Gemini explicitly rejects tool-role messages in this PR.

Cloudflare positions GLM-4.7-Flash specifically for **multi-turn tool calling**, confirming that continuation behavior is part of the intended workflow. ([Cloudflare Docs][1])

### Required detection

```ts
const hasToolHistory = req.messages.some((message) => {
  if (message.role === "tool") return true;

  return (
    message.role === "assistant" &&
    (message.tool_calls?.length ?? 0) > 0
  );
});

const hasDeclaredTools = (req.tools?.length ?? 0) > 0;
const needsToolProvider = hasDeclaredTools || hasToolHistory;
```

### Missing architectural improvement

Do not depend only on reconstructing the tier from messages. Also consider storing the selected provider/model in conversation metadata so every continuation uses the same provider unless an explicit migration occurs.

# 4. Tool model verification

## Llama Fast

The uploaded report is too strong when it says the model lacks all documented tool support.

Cloudflare’s Llama Fast page confirms:

* model is active;
* 128,000-token context;
* streaming;
* synchronous output includes `tool_calls`. ([Cloudflare Docs][2])

However, its model information does **not** explicitly show the same clear:

```text
Function calling: Yes
```

label shown for GLM-4.7-Flash. Therefore, the model is less strongly documented as the production tool tier.

More importantly, the internal registry says:

```ts
capabilities: ["text", "streaming"]
```

while assigning it to the `tool-calling` tier. That is internally contradictory.

## Better model choice

`@cf/zai-org/glm-4.7-flash` is the strongest immediate choice based on official documentation:

* explicit function calling;
* explicit reasoning;
* multi-turn tool calling;
* 131,072-token context;
* $0.06/M input;
* $0.40/M output;
* optimized for dialogue and 100+ languages. ([Cloudflare Docs][1])

### Recommended registry entry

```ts
"tool-calling": {
  provider: "workers-ai",
  model: "@cf/zai-org/glm-4.7-flash",
  capabilities: [
    "text",
    "streaming",
    "function-calling",
    "reasoning",
  ],
  contextWindow: 131072,
  costPer1kIn: 0.00006,
  costPer1kOut: 0.0004,
},
```

Do not use `"structured"` as a replacement label for `"function-calling"` unless your internal capability model intentionally equates those two concepts. They are not the same feature.

# 5. Pricing error

The current Llama registry entry records:

```ts
costPer1kIn: 0.00015,
costPer1kOut: 0.00015,
```

The Cloudflare model page inspected here does not establish those exact values. Therefore, the registry contains an unsupported pricing assumption.

This affects:

* cost telemetry;
* provider routing;
* budget enforcement;
* model comparisons;
* production cost estimates.

### Required fix

Either:

* source exact official pricing for that exact model identifier; or
* use `null`/`undefined` and label pricing as unresolved.

Do not store invented or inherited pricing as fact.
