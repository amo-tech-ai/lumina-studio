# What is correct

## 🟢 1. OpenAI-compatible architecture

Using an OpenAI-compatible interface between Mastra and the Gateway Worker is sound.

Cloudflare officially supports:

* `/v1/chat/completions`
* `/v1/embeddings`
* standard OpenAI-shaped requests
* changing the base URL and model rather than rewriting callers

Cloudflare also documents that these endpoints can work with AI Gateway. ([Cloudflare Docs][1])

The planned flow is therefore reasonable:

```text
Mastra
→ OpenAI-compatible model client
→ custom Gateway Worker
→ Workers AI / Gemini
```

---

## 🟢 2. AC-F before full Mastra migration

This separation is correct:

* **IPI-454 · CF-AI-001 — AI Gateway — Cloudflare Provider Routing** AC-F proves one shared gateway model path.
* **IPI-485 · MASTRA-CF-001 — Mastra Provider Gateway Cutover** migrates all agents later.

That limits blast radius and avoids changing every agent in one PR.

---

## 🟢 3. AC-J as a separate verification gate

Keeping implementation and complete end-to-end proof as separate stages is good.

AC-J should verify:

* routing
* streaming
* structured responses
* tools
* failure behavior
* telemetry

That is stronger than claiming success from unit tests alone.

---

## 🟢 4. Production deployment remains separate

Keeping production deployment in:

**IPI-472 · INFRA-001 — Cloudflare Worker Deployment Pipeline**

is correct.

OpenNext clearly distinguishes:

* local Workers preview;
* upload of a Worker version;
* actual Cloudflare deployment.

`npm run preview` is local, while `upload` and `deploy` are remote actions. ([OpenNext][2])

## 🔴 3. IPI-461 completion boundary is unclear

The issue treats **IPI-461 · CF-AI-004 — AI Provider Adapter** as a prerequisite for AC-F, but the adapter currently works for:

* chat
* stream
* structured output

while embeddings fail.

You need an explicit decision:

### Option A — AC-F is chat-only

Proceed with AC-F now, while keeping IPI-461 open for embeddings.

### Option B — all adapter methods required

Fix embeddings first, then start AC-F.

For the current Mastra/CopilotKit cutover, **Option A is likely preferable**, provided no migrated caller needs embeddings. This must be documented rather than assumed.

---

## 🔴 4. The target architecture risks duplicating adapter layers

The plan says:

```text
resolveModel()
→ @ai-sdk/openai-compatible
→ AI_GATEWAY_URL
```

But the code also has a custom `providerAdapter`.

These are two potentially overlapping clients:

1. AI SDK model provider used by Mastra.
2. Custom REST adapter used by health, chat, structured and embedding calls.

### Required architecture decision

Define ownership clearly:

| Use case                       | Client                                            |
| ------------------------------ | ------------------------------------------------- |
| Mastra agent `model` interface | `@ai-sdk/openai-compatible`                       |
| Health probe                   | `createProviderAdapter()` or direct health client |
| Non-Mastra service calls       | `providerAdapter`                                 |
| Embeddings                     | One chosen client only                            |

Do not make `resolveModel()` wrap the custom adapter unless it satisfies Mastra/AI SDK’s language-model interface.

The custom adapter and AI SDK provider can coexist, but only with documented boundaries.

---

## 🔴 5. AC-J tool calling is not guaranteed by OpenAI compatibility alone

Cloudflare supports OpenAI-compatible endpoints, but that does not mean every Workers AI model supports every OpenAI feature.

Tool calling depends on:

* the exact selected model;
* Workers AI model capability;
* Gateway request forwarding;
* Mastra tool schema compatibility;
* streamed tool-call delta handling.

Cloudflare separately labels function calling as model/platform functionality, and some features remain model-specific. ([Cloudflare Docs][1])

### Critical fix

Add model-specific probes:

* selected model supports tool use;
* non-stream tool call works;
* streamed tool call works;
* tool arguments remain valid JSON;
* tool results return to Mastra;
* CopilotKit receives the expected AG-UI events.

# Red flags and likely failure points

## 🟡 1. App and Worker registries remain divergent

The plan claims a unified model registry, but the app and Gateway Worker still have separate registry shapes.

This can produce:

* tier name drift;
* different default models;
* incorrect capability declarations;
* chat selecting one model while the Worker routes another.

### Improvement

Before or inside AC-F, establish one contract for:

```ts
type ModelTier =
  | "default"
  | "fast"
  | "structured"
  | "vision"
  | "embedding";
```

The Worker may keep its own runtime registry, but tier IDs and capability semantics must be shared or tested against each other.

---

## 🟡 2. The model registry says Workers AI is default before evaluation

The registry comments reportedly label Workers AI as the MVP default, while the roadmap says the default tier flip is blocked until:

**IPI-462 · CF-AI-006 — AI Provider Evaluation Suite**

Those statements conflict.

### Improvement

Use separate concepts:

* `configuredDefault`
* `candidateDefault`
* `productionDefault`

Until evaluation passes, production traffic should remain on the currently approved provider or use a controlled feature flag.

---

## 🟡 3. Fallback scaffolding is not the same as verified fallback

AC-D is marked complete as “retry/fallback scaffolding.”

But AC-J expects actual fallback behavior, while **IPI-463 · CF-AI-008 — AI Provider Failover & Rollback** owns circuit breakers and failover policy.

### Better distinction

* AC-D: provider code can attempt fallback.
* AC-J: one controlled failure proves fallback routing.
* IPI-463: production retry policy, circuit breaker, rollback and provider health state.

Do not mark production failover complete from basic scaffolding.

---

## 🟡 4. AC-J metrics wording is ambiguous

The checklist says:

> Metrics visible inside Gateway dashboard.

Your architecture includes a **custom Gateway Worker**, not only Cloudflare’s managed AI Gateway.

Metrics may live in:

* Worker observability;
* AI Gateway analytics;
* custom logs;
* Sentry;
* request-level telemetry.

### Improvement

Specify exact required evidence:

| Metric             | Source                    |
| ------------------ | ------------------------- |
| request count      | Worker logs or AI Gateway |
| latency            | Worker observability      |
| provider selected  | structured Worker log     |
| status/error class | structured Worker log     |
| fallback count     | custom metric/log         |
| model/tier         | sanitized metadata        |
| trace/request ID   | response header and logs  |

---

## 🟡 5. Authentication design is incomplete

AC-F says configure `AI_GATEWAY_URL`, but it needs to define how the application authenticates to the Gateway Worker.

Required questions:

* Is the Gateway Worker public?
* Is `AI_GATEWAY_API_KEY` required?
* Is a Cloudflare Service Binding preferred for Worker-to-Worker calls?
* Are secrets available in Vercel and OpenNext preview?
* Are preview and production credentials separate?
* Does the health endpoint require authentication?

For two Workers inside Cloudflare, a Service Binding can avoid public internet routing, but that is an architecture choice—not automatically required.

---

## 🟡 6. Environment parity is missing

AC-F may pass in `next dev` and fail in OpenNext because environment delivery differs.

OpenNext requires Wrangler configuration and Cloudflare bindings for deployment; local preview runs in the Workers runtime, while normal Next development remains a different path. ([OpenNext][2])

Add a matrix:

| Variable             | Next dev | OpenNext preview | Remote preview | Production |
| -------------------- | -------- | ---------------- | -------------- | ---------- |
| `AI_GATEWAY_URL`     |          |                  |                |            |
| `AI_GATEWAY_API_KEY` |          |                  |                |            |
| default model tier   |          |                  |                |            |
| fallback provider    |          |                  |                |            |

---

## 🟡 7. Health does not prove inference

`GET /api/ai/health` only proves:

* application can resolve the URL;
* Gateway Worker responds;
* adapter can be constructed.

It does not prove:

* model credentials;
* provider availability;
* chat completion;
* streaming;
* tool calling;
* structured output.

Keep health as a readiness signal, but add an authenticated synthetic inference probe for staging.

# Missing items

## 1. Explicit AC-F Definition of Done

Add:

* one Mastra agent calls `resolveModel("default")`;
* generated model uses `AI_GATEWAY_URL`;
* Gateway logs confirm receipt;
* no direct Gemini/Groq SDK call for that path;
* streaming reaches CopilotKit;
* cancellation propagates;
* gateway failure gives a controlled application error;
* legacy path remains behind an explicit rollback flag.

---

## 2. Rollback flag

Before cutting over Mastra, add an explicit rollback mechanism such as:

```text
AI_ROUTING_MODE=direct
AI_ROUTING_MODE=gateway
```

or an equivalent feature flag.

Do not require a code revert to restore the old provider path.

---

## 3. Correlation IDs

Add a request/trace ID through:

```text
CopilotKit
→ Mastra
→ Gateway Worker
→ provider
```

Without it, debugging AC-J failures will be difficult.

---

## 4. Streaming protocol verification

Test more than token delivery:

* `[DONE]`;
* partial chunk boundaries;
* malformed SSE line;
* tool-call delta;
* provider closes without `[DONE]`;
* client cancellation;
* idle timeout;
* downstream 429/500.

---

## 5. Security gate before production

**IPI-468 · SEC-001 — Cloudflare AI Security Architecture** should not necessarily block local AC-F development, but it should block production AC-I unless the required security subset is complete:

* fail-closed authentication;
* secrets;
* rate limits;
* allowed callers;
* logging redaction;
* preview access controls.

---

## 6. Cost controls

Before production routing, add:

* maximum tokens;
* model allowlist;
* request timeout;
* per-org/user rate limit;
* daily budget alert;
* provider usage metadata.
