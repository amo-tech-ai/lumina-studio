## The simple version of what I compared

I looked at 8 different ways to handle AI requests on Cloudflare, from "write everything yourself" to "let Cloudflare's dashboard handle it." The clear winner:

**Use Cloudflare's AI Gateway** — a dashboard feature that already does security checks, automatic retries, and switching between AI providers, all without writing code for it. Pair that with a small, Cloudflare-published connector package instead of hand-rolled adapters for each AI provider (Gemini, Bedrock, etc.).

The one option that's clearly the *wrong* fit: Cloudflare's "Agents" starter kit. That's built for AI chatbots that remember conversations over time — iPix's setup just answers one request and moves on, so that toolkit would add complexity for a problem iPix doesn't have.

## How to finish this using the Cloudflare dashboard

Good news: most of what's left is dashboard clicking, not writing code. Here's the order, each one pulled from the existing plan documents (not invented fresh) — the plan already has these written out in full detail:

### Step 1 — Create the AI Gateway (15 minutes, dashboard only)
1. Log into the Cloudflare dashboard.
2. Left sidebar → **AI** → **AI Gateway**.
3. Click **Create Gateway**, name it `ipix-prod`.
4. Click **Create**. Done — this is a one-time setup, no code involved.

*What this replaces:* the custom routing code (~400 lines), the custom retry logic (~112 lines), and manual cost-tracking that doesn't currently exist.

### Step 2 — Turn on automatic retries (5 minutes, dashboard only)
1. Open the gateway you just created → **Settings** (or **Request Handling**).
2. Turn on **Auto-Retry**.
3. Set max attempts to 3, delay to 1000ms.
4. Save.

*What this does:* if an AI provider briefly hiccups, Cloudflare automatically tries again — the same thing the custom retry code does today, minus the risk of bugs in homemade retry logic.

### Step 3 — Turn on automatic fallback to a backup model (15 minutes, dashboard only)
1. In the gateway, open the **Dynamic Routing** tab.
2. Click **Create route**.
3. Set your everyday model as "primary" and a backup model as "fallback."
4. Choose what triggers the fallback (a server error, a timeout, or running out of budget).
5. Save.

*What this does:* if your main AI provider goes down, Cloudflare automatically switches to the backup one — this is the single biggest chunk of custom code it replaces (the router, the retry classifier, and the error-mapping code all get deleted once this is verified working).

### Step 4 — Add the AI binding to the code (15 minutes, one small code change — not dashboard)
This one small step needs a one-line code edit, not a dashboard click: add an `ai` entry to the project's `wrangler.jsonc` configuration file, so the app can talk to Workers AI directly. Full instructions with the exact line to add are already written out in `tasks/cloudflare/Tasks/003-CF-AI-add-workers-ai-binding.md`.

### Step 5 — Turn on the security check natively (once the above is working)
Once requests are flowing through the AI Gateway, turn on its built-in **Authenticated Gateway** toggle in the dashboard instead of relying on the hand-written bearer-token check. One real caveat to know about first: Cloudflare's gateway tokens work per Cloudflare *account*, not per individual gateway — so this is a slightly different security model than the current custom check, worth confirming that trade-off is acceptable before switching.

### Step 6 — Delete the old custom code (only after Steps 1–5 are proven working)
Once everything above is confirmed working, the old hand-written gateway code gets deleted — about 2,300 lines across the project. This step is already fully documented in `tasks/cloudflare/Tasks/053-CF-MIGRATION-cleanup-custom-code.md`, including the exact files to remove and how to roll back if anything breaks.
