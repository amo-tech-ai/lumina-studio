# IPI-622 · CF-DB-008 — Hyperdrive vs Supabase Data API vs Placement benchmark

**Date:** 2026-07-22
**Status:** Benchmark complete. Does not gate app behavior — informs IPI-623 (migrate one Mastra
workload to Hyperdrive) and IPI-624 (alert thresholds).

## What was measured, and how

A throwaway benchmark Worker (`scripts/bench/ipi-622-hyperdrive-benchmark/`) was deployed twice —
once with no `placement` block, once with `"placement": {"mode": "smart"}` — both bound to the
**real preview Hyperdrive config**, not a mock:

| Item | Value |
|---|---|
| Hyperdrive config | `ipix-supabase-fresh` (id `f59421821941436593f4c88416fb1601`) |
| Caching | Disabled (confirmed via `hyperdrive_config_get` — this is the "fresh" config, not cached) |
| Origin connection limit | 5 |
| Worker (no placement) | `ipix-bench-hyperdrive` → `https://ipix-bench-hyperdrive.sk-498.workers.dev` |
| Worker (Smart Placement) | `ipix-bench-hyperdrive-placement` → `https://ipix-bench-hyperdrive-placement.sk-498.workers.dev` |
| Runner | `scripts/bench/ipi-622-hyperdrive-benchmark/run-bench.mjs`, 40 sequential requests per route, 3 discarded warmup requests per route |
| DB tables (Hyperdrive side) | `mastra.mastra_threads`, `mastra.mastra_messages`, `mastra.mastra_resources` (IPI-628 schema, empty/test-safe, granted to `hyperdrive_mastra_runtime` per IPI-629) |
| DB table (Data API side) | `public.brands`, read-only `SELECT id,name,brand_url LIMIT 20` via service-role key (RLS on `brands` is org-scoped `authenticated`-only, so anon key can't read it — service role was used for benchmarking only) |

Each Worker exposes 4 routes measured both **externally** (full HTTP round trip as seen by the
runner script) and **internally** (the Worker's own `performance.now()` timers around just the
`pg`/`supabase-js` call — this is the fairer number for comparing Hyperdrive vs Data API, since it
excludes this-machine-to-edge network noise):

- `/select` — simple SELECT, `mastra.mastra_threads`, `LIMIT 20`
- `/join` — 3-table LEFT JOIN, `mastra.mastra_threads` ⋈ `mastra.mastra_messages` ⋈ `mastra.mastra_resources`
- `/insert-select` — INSERT a bench row into `mastra.mastra_threads`, SELECT it back, then DELETE it (keeps the table empty for later Hyperdrive tasks — only `mastra` schema rows were ever mutated, no `public.*` product data touched)
- `/dataapi` — Supabase Data API equivalent of `/select`, against `public.brands`

Full raw output: `2026-07-22-fresh-no-placement.json`, `2026-07-22-placement-smart.json` (this directory).

**Caveat on absolute numbers:** the `mastra.*` tables are intentionally empty (per ticket scope),
so these queries measure connection/round-trip/planning overhead, not large-table scan cost. That's
still the right thing to measure here — the question this ticket answers is "which network path is
cheaper from a Worker," not "how does Postgres scale with data volume."

## Results — Hyperdrive fresh vs Supabase Data API (internal query time, ms)

| Route | Path | P50 | P95 | P99 |
|---|---|---:|---:|---:|
| `/select` | Hyperdrive | 43 | 62 | 128 |
| `/select` | Data API | 51 | 61 | 354 |
| `/join` | Hyperdrive | 43 | 44 | 60 |
| `/join` | Data API | — (not run on Data API; see note below) | | |
| `/insert-select` | Hyperdrive | 128 | 161 | 175 |
| `/insert-select` | Data API | — (Data API leg is read-only per ticket scope; no INSERT test on `public.brands`) | | |

Note: the multi-join and insert+select legs are Hyperdrive-only (they target the `mastra` schema
IPI-628 tables specifically). The Data API comparison is the "first," simple-select comparison the
ticket calls for; a matching join was skipped since `public.brands` has no benchmark-safe joinable
table wired up, and adding one would be scope creep for a Low-priority benchmarking ticket.

**Reading:** Hyperdrive is marginally faster at P50 (43ms vs 51ms) and roughly comparable at P95;
Data API had one P99 outlier (354ms, single sample of 40) that looks like a one-off cold path
rather than a systemic difference. **Neither path "clearly" wins** — per the ticket's own
instruction ("only add a cached-config comparison if fresh clearly proves value"), this result does
**not** justify spending time on a cached-Hyperdrive-config comparison. Skipped; revisit only if a
real workload shows Hyperdrive fresh consistently losing to Data API under load.

## Results — Placement A/B (internal query time, ms)

| Route | Variant | P50 | P95 | P99 |
|---|---|---:|---:|---:|
| `/select` | no placement | 45 | 64 | 129 |
| `/select` | smart placement | 47 | 55 | 73 |
| `/join` | no placement | 45 | 47 | 61 |
| `/join` | smart placement | 50 | 55 | 55 |
| `/insert-select` | no placement | 129 | 162 | 176 |
| `/insert-select` | smart placement | 136 | 155 | 210 |

**Reading: no evidence Smart Placement helps.** P50 is essentially flat to slightly worse with
placement enabled across all three routes; `/insert-select` P50 is 7ms worse. The external
(client-to-Worker) round trip told the same story — 105–110ms P50 for both variants, well within
noise for a single-region test client (both hit the Miami colo, `cf-ray: …-MIA`).

This is consistent with Cloudflare's own Smart Placement docs, which describe it as a traffic-shape
optimization that needs a sustained real workload to activate — a synthetic 40-request burst
against a freshly-deployed Worker is exactly the case it isn't designed to prove out. **Per the
ticket's explicit instruction, Placement is NOT recommended to be enabled by default based on this
evidence.** If it's revisited, it should be measured against real, sustained traffic (e.g. during
IPI-623's actual Mastra workload migration), not a synthetic script.

## Reproducing this benchmark

```bash
cd scripts/bench/ipi-622-hyperdrive-benchmark
npm install
# Cloudflare + Supabase creds are read from app/.env.local by the caller; see the deploy commands
# actually used in this session (CLOUDFLARE_ACCOUNT_ID / CLOUDFLARE_API_TOKEN env, then):
npm run deploy              # no-placement variant
npm run deploy:placement    # smart-placement variant
# then set secrets once per Worker:
#   wrangler secret put SUPABASE_URL -c wrangler.jsonc
#   wrangler secret put SUPABASE_SERVICE_ROLE_KEY -c wrangler.jsonc
#   (repeat with -c wrangler.placement.jsonc)
npm run bench -- --url <deployed-url> --label <name> --n 40
```

## Cleanup

Both benchmark Workers (`ipix-bench-hyperdrive`, `ipix-bench-hyperdrive-placement`) were deleted
after this run — they held a `SUPABASE_SERVICE_ROLE_KEY` secret (needed to read `public.brands`
past RLS for the Data API leg) and had no reason to stay deployed once the numbers were collected.
Rerun via the commands above if the numbers need refreshing.

## Dependency note

This benchmark's Worker config (`scripts/bench/ipi-622-hyperdrive-benchmark/wrangler*.jsonc`) is
**fully standalone** — it points at the same Hyperdrive config id (`f59421821941436593f4c88416fb1601`)
that `app/wrangler.jsonc`'s `HYPERDRIVE_FRESH` binding uses (added by IPI-619 / PR #600, still open
as of this writing), but does not depend on that PR merging, since multiple Workers can bind to one
Hyperdrive config. **This PR's diff does not touch `app/wrangler.jsonc`.**

## Summary for IPI-623 / IPI-624

- Hyperdrive fresh (no cache) is roughly on par with the Supabase Data API for simple reads from a
  Worker — pick based on developer ergonomics (raw SQL vs PostgREST) and RLS needs, not latency.
- Do not enable Smart Placement by default — no measured benefit, one route measured slightly worse.
- Suggested alert thresholds for IPI-624 (based on P95 observed here, with headroom): Hyperdrive
  simple query P95 alert > 150ms, insert+select P95 alert > 250ms. These are starting points from a
  40-sample synthetic run against empty tables, not load-tested production thresholds — revisit
  once IPI-623 puts a real workload on Hyperdrive.
