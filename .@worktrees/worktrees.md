## Lifecycle

Every worktree in this repo should pass through the same five stages. A row stuck outside "Work" for more than a few days is what the health/dirty/unpushed columns below are for catching.

1. **Create** тАФ `npm run worktree:add -- IPI-NNN slug` (or `EnterWorktree`/`claude --worktree`). Always branches from a freshly fetched `origin/main`, never from a stale local ref.
2. **Work** тАФ commit early, push often. An uncommitted worktree is one `rm -rf` away from losing everything; an unpushed one is one `git branch -D` away.
3. **Merge** тАФ PR merges on GitHub. The worktree itself doesn't know this happened yet тАФ its local branch still looks "ahead" until step 4.
4. **Sync** тАФ `npm run worktree:sync-main` fast-forwards local `main`. Refuses instead of forcing if `main` ever has commits `origin/main` doesn't (a real divergence, not just staleness).
5. **Cleanup** тАФ `npm run worktree:pre-delete` (or trust the "Safe to delete now" list below, which already runs the same check), then `git worktree remove` + `git worktree prune` + `git branch -D` (not `-d` тАФ this repo squash-merges, so a merged branch's commit history is never a literal ancestor of `main`; `-D` is only safe here because `deletable` already verified the PR is MERGED, not because ancestry says so).
